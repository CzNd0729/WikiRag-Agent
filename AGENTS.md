# AI Agent 辅助编程行为规范 (AGENTS.md)

为了确保本项目的代码质量、工程化标准以及开发协作的高效性，所有参与本项目的 AI Agent 必须严格遵守以下开发规范：

## 🛠 1. 软件工程原则

*   **高内聚，低耦合**: 在编写、重构或拆分代码时，务必遵循模块化设计原则。每个模块/函数应仅负责一项功能，减少不同组件间的直接依赖。
*   **DRY (Don't Repeat Yourself)**: 避免重复代码，优先考虑提取公共工具类或组件。
*   **代码自解释**: 变量与函数命名应具有清晰的语义，复杂逻辑需配有简洁的注释。

## 📋 2. 任务管理与进度追踪

*   **强制列出 TODO**: 在每次开始任务或进入新阶段时，必须通过 `task_progress` 显式列出任务清单（Checklist）。
*   **及时更新状态**: 
    *   完成一个子任务后，立即更新清单状态（标记为 `[x]`）。
    *   如发现任务范围发生变化，需及时调整清单内容。
*   **阶段性汇报**: 在完成重要里程碑后，应向用户简要汇报当前进度。

## 📁 3. 项目环境维护

*   **及时维护 .gitignore**: 
    *   在引入新依赖、生成临时文件、缓存或敏感信息（如 API Key）时，必须检查并更新 `.gitignore`。
    *   严禁将 `.env`、虚拟环境目录（如 `venv/`）或操作系统缓存（如 `.DS_Store`）提交到版本控制。
*   **依赖管理**: 新增库文件时，需同步更新 `pyproject.toml`。

## 📖 4. 上下文感知

*   **必须阅读 README.md**: 在执行任何功能开发前，Agent 必须先阅读 `README.md`，以确保理解项目背景、技术方案及现有的工具集成（如 MCP Server 配置）。
*   **遵守既定模式**: 严禁随意引入与项目既定架构冲突的技术栈或开发模式。

## 💬 5. 沟通与交互规范

*   **客观与诚实**: 在回复时应保持技术客观性。如果某个方案存在局限性或风险（如 API 额度超限风险），必须提前告知用户。
*   **疑虑必问**: 
    *   如果用户指令存在歧义，禁止猜测，必须及时提问澄清。
    *   遇到环境配置、Key 缺失或权限问题时，应第一时间向用户确认，不要盲目重试。
*   **精简回复**: 在 Act Mode 下，减少冗长的开场白和解释，优先展示工具执行结果。
*   **测试义务**: 除非用户特别明确要求 Agent 进行代为测试（例如“帮我跑一下测试”），否则 Agent **无需主动**使用 `execute_command` 或 `browser_action` 进行功能验证。应直接在回复中清晰地告知用户**测试入口命令**以及**预期的执行结果**，由用户自行决定是否运行。

## 🧪 6. 工具使用偏好

*   **优先使用专用工具**: 在读取、更改或搜索项目文件时，必须优先使用内置的 `read_file`、`write_to_file`、`replace_in_file` 或 `search_files` 等专用工具。
*   **限制命令行使用**: 除非现有工具无法满足需求（如安装依赖、启动服务等），否则严禁使用 `execute_command` 进行文件读写或内容检索，以确保操作的原子性和可追溯性。
*   **结果校验**: 确保生成的文件符合预期格式，逻辑严密，且不产生冗余副本。

## 🏗 7. 项目结构规范

*   **实时维护 README**: 在重构或新增模块后，必须立即更新 `README.md` 中的“项目结构”章节。
*   **结构合理性**: 代码组织应遵循业务逻辑而非工具类别。例如，核心逻辑应置于 `src/core/`，具体业务 Agent 定义置于 `src/agents/`，确保层级清晰。
*   **导入规范**: 在引入本地模块代码包时，**严禁使用 `src.` 前缀**。应直接使用顶级包名进行导入（例如：`from core.rag_engine import ...` 而非 `from src.core.rag_engine import ...`），以确保代码在作为包安装后的路径兼容性。

*本规范旨在建立 Agent 与人类开发者之间的技术默契，持续维护项目的高标准交付。*
